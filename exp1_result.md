# 《操作系统》课程实验报告

## 一、 实验目的与题目
本实验选自“实验一：进程间同步/互斥问题” 中的“银行柜员服务问题”。
本实验的主要目的是通过编程实现该问题，使用P、V操作来实现柜员和顾客之间的同步，以加深对操作系统中并发控制、互斥和同步机制的理解，并熟练掌握相关函数的应用。

程序严格遵循了以下实现要求：
1. 某个号码只能由一名顾客取得。
2. 不能有多于一个柜员叫同一个号。
3. 有顾客的时候，柜员才叫号。
4. 无柜员空闲的时候，顾客需要等待。
5. 无顾客的时候，柜员需要等待。

## 二、 设计思路与程序结构

### 1. 并发控制设计思路
本程序采用多线程模拟顾客和柜员的并发行为，基于互斥和同步机制来调度共享资源：
* **互斥对象**：顾客拿号操作、柜员叫号操作（即对等待队列和号码分配的访问）需要互斥。程序中通过初始化值为1的信号量 `queue_mutex` 来保护共享的等待队列 `waiting_queue` 和全局变量 `next_number`，确保所有数据结构在访问时也需要互斥。此外，为防止控制台输出错乱，增加了 `cout_mutex` 互斥锁。
* **同步对象**：顾客和柜员构成了主要的同步关系。
    * **柜员等待顾客**：使用初始值为0的信号量 `customer_count`。顾客取号后执行V操作，唤醒柜员；柜员在服务前执行P操作，无顾客时则阻塞等待。
    * **顾客等待服务完成**：在 `Customer` 结构体内部维护了一个局部信号量 `finished`（初值为0）。柜员完成服务后对该顾客的信号量执行V操作，而顾客在进入系统后对其执行P操作，实现自身的阻塞等待直到服务完毕。

### 2. 程序结构
* **`Semaphore` 类**：基于 C++11 标准库的 `std::mutex` 和 `std::condition_variable` 封装的自定义信号量类，提供标准的 `P()`（等待/消费）和 `V()`（唤醒/生产）操作。
* **`Customer` 结构体**：存储每位顾客的 ID、到达时间、服务时间以及进入、开始、离开的具体时间戳，内部包含专属同步信号量 `finished`。
* **`customer_thread` 线程函数**：模拟顾客到达、互斥获取号码并进入队列、唤醒柜员等待服务完成并最终离开的生命周期。
* **`teller_thread` 线程函数**：模拟柜员循环工作。通过P操作等待顾客，获取到顾客后互斥出队，休眠指定的 `service_time` 模拟服务过程，结束后唤醒顾客。
* **`main` 主函数**：负责解析输入数据（柜员数及测试文本）、初始化并派发多线程、通过注入 `nullptr` 唤醒并终结柜员线程，最后格式化输出结果。

## 三、 关键代码分析

### 1. 信号量的实现
通过条件变量和互斥锁实现 P/V 操作，保证了线程安全的阻塞与唤醒，避免了忙等待浪费 CPU 资源。
```cpp
class Semaphore {
private:
    mutex mtx;
    condition_variable cv;
    int count;
public:
    explicit Semaphore(int init_count = 0) : count(init_count) {}
    void P() {
        unique_lock<mutex> lock(mtx);
        cv.wait(lock, [&]() { return count > 0; });
        count--;
    }
    void V() {
        unique_lock<mutex> lock(mtx);
        count++;
        cv.notify_one();
    }
};
```

### 2. 顾客进入系统与取号 (互斥与同步)
顾客进入银行后，首先需要互斥地获取号码并加入队列，随后通过 `customer_count.V()` 唤醒可能在休眠的柜员，接着在自身的 `finished` 信号量上阻塞，等待服务完成。
```cpp
queue_mutex.P(); // 互斥锁：保护全局号码和队列
customer->number = next_number++;
waiting_queue.push(customer);
queue_mutex.V();

customer_count.V(); // 同步：增加队列中顾客数量，唤醒柜员
customer->finished.P(); // 同步：阻塞自己，等待柜员办理完成
```

### 3. 柜员处理逻辑与线程安全退出
柜员持续监听 `customer_count`。当取到 `nullptr` 时，意味着主线程已判定所有顾客服务完毕并派发了“下班”信号，柜员跳出循环结束线程。
```cpp
customer_count.P(); // 同步：请求等待队列，无顾客则等待
queue_mutex.P();    // 互斥：从等待队列中取出顾客
Customer* customer = waiting_queue.front();
waiting_queue.pop();
queue_mutex.V();

if(customer == nullptr) { // 优雅结束线程的巧妙设计
    print_event("柜员 " + to_string(teller_id) + " 下班");
    break;
}
// ... 模拟服务 ...
customer->finished.V(); // 同步：完成服务，唤醒对应的顾客
```

## 五、 思考题解答

**1. 柜员人数和顾客人数对结果分别有什么影响？**
* **柜员人数的影响**：柜员人数决定了系统并发处理的上限（即吞吐量）。当柜员人数较少时，容易形成排队积压，导致顾客的等待时间（`Start` 减去 `Enter`）变长；增加柜员人数可以有效降低顾客的平均等待时间。然而，当柜员人数大于等于系统内并发出现的最高顾客数时，继续增加柜员将无法进一步优化速度，反而会增加操作系统在线程调度和上下文切换上的开销。
* **顾客人数的影响**：顾客人数（特别是短时间内的到达率）构成了系统的负载压力。顾客到来频率越高，等待队列越长。庞大的顾客基数会直接考验互斥锁的竞争性能和队列数据结构的内存稳定性。

**2. 实现互斥的方法有哪些？各自有什么特点？效率如何？**
* **硬件指令法（如 TestAndSet、Swap指令或中断屏蔽）**：
    * *特点*：利用底层硬件指令的原子性实现。逻辑简单，适用于多处理器。
    * *效率*：低。失败的线程会陷入“忙等待（自旋）”，持续占用 CPU 资源，可能导致饥饿现象。中断屏蔽更是无法适应现代多核处理器。
* **软件算法（如 Peterson 算法、Dekker 算法）**：
    * *特点*：纯软件层面利用标志位实现互斥。
    * *效率*：较低。不仅编写和扩展到N个进程极为复杂，且同样存在忙等待（Busy Waiting）的问题。
* **操作系统级机制（如 信号量 Semaphore、互斥锁 Mutex）**：
    * *特点*：由操作系统内核或语言运行库（如 `std::mutex`）提供支持。当资源不可用时，线程会主动让出 CPU 进入睡眠/阻塞状态（让权等待），并在资源可用时被唤醒。
    * *效率*：高。避免了无意义的自旋消耗，极大地提高了 CPU 的利用率，是现代操作系统和实际工程中最主流、最高效的互斥实现手段。

## 六、 实验体会与遇到的问题
在本次实验中，我遇到了多线程并发输出时的“控制台竞争”问题，导致初期运行的终端打印日志存在字符穿插和换行错乱的现象。通过分析，我意识到不仅业务数据结构需要互斥，标准输出流（`std::cout`）同样是共享资源。随后我额外增加了一个互斥锁 `cout_mutex`，彻底解决了画面撕裂问题，加深了对“万物皆需考虑并发安全”的认识。

同时，在处理柜员线程退出机制时，如何让一直阻塞在 `customer_count` 信号量上的柜员安全结束是一个难点。我没有采用强制的 `thread::detach()` 或是暴力中断，而是主线程在确认所有顾客都处理完后，向等待队列注入了相当于柜员数量的 `nullptr`（毒丸模式），再配合 V 操作唤醒所有柜员。这种优雅的关闭方式让整个程序的生命周期管理变得异常清晰，也让我极大提升了对 C++ 多线程编程范式的掌握。
